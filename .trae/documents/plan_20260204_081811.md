# 修复方案：修正排班 DTO 校验规则

## 问题根因
1.  **Server 日志报错**：`EADDRINUSE: address already in use :::3000`，说明 Server 端口被占用。这是因为之前的 Server 进程可能未正确退出，而我尝试启动新进程导致冲突。
2.  **Web 400 错误**：用户反馈的排班创建 400 错误 (`Schedule creation failed AxiosError: Request failed with status code 400`)，经过代码分析：
    -   前端 `ScheduleDialog.tsx` 发送的数据中，`employeeId` 和 `shiftId` 经过 `Number()` 转换。
    -   后端 `CreateScheduleReqDto` 使用 `class-validator` 的 `@IsInt()` 装饰器。
    -   **关键点**：`plainToInstance` 默认不会自动将字符串转换为数字，除非开启 `enableImplicitConversion` 选项。如果前端传递的是 JSON 数字类型，这通常没问题。但如果中间件或前端逻辑有变（例如表单值默认是字符串），可能导致校验失败。
    -   更可能的根因是：前端 `ScheduleDialog.tsx` 中 `force` 字段默认是 `false`，但后端 `CreateScheduleReqDto` 中 `force` 是 `@IsBoolean()` 和 `@IsOptional()`。如果前端传递了 `force: false`，这是符合校验的。
    -   **真正的校验坑**：`CreateScheduleReqDto` 中的 `startDate` 和 `endDate` 使用了 `DATE_REGEX` 正则校验 (`/^\d{4}-\d{2}-\d{2}$/`)。如果前端传递的日期格式带有时间部分（如 ISO 字符串），或者前端 `type="date"` 的输入值在某些浏览器下表现不一致，就会导致正则校验失败，抛出 400 错误。

## 实施步骤
1.  **解决端口占用**：
    -   先停止所有占用 3000 端口的进程。
    -   重启 Server 服务。
2.  **优化后端 DTO**：
    -   修改 `packages/server/src/modules/attendance/schedule/schedule.dto.ts`。
    -   移除严格的正则校验 `@Matches(DATE_REGEX)`，改用 `class-transformer` 的 `@Type(() => Date)` 和 `class-validator` 的 `@IsDate()`，或者保留正则但确保前端传递正确格式。考虑到稳定性，建议后端放宽接收格式（支持 ISO 字符串），然后在 Service 层统一处理。
    -   **更稳妥方案**：保持正则校验，但在 `ScheduleController` 中使用 `plainToInstance` 时启用 `{ enableImplicitConversion: true }`，确保数字类型被正确转换。
3.  **验证**：
    -   重启服务后，在 Web 端再次尝试创建排班。

## 验证标准
- Server 正常启动无端口冲突。
- 排班创建成功，不再返回 400。
