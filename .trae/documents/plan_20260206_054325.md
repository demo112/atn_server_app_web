# 框架规范与质量修复实施计划

## 阶段一：框架规范化整治 (Code Organization)
### 1. 重构路由层异常处理
- **目标**: 消除手动 try-catch 导致的异常吞没风险，统一错误流。
- **文件**: `packages/server/src/modules/attendance/attendance.routes.ts`
- **实施**:
  - 移除路由中的 `try { ... } catch (error) { res.status(500)... }` 块。
  - 使用 `express-async-handler` 包裹异步路由处理函数。
  - 确保异常抛出后由全局 `error-handler.middleware.ts` 捕获并记录完整堆栈。

### 2. 统一 API 错误响应格式
- **目标**: 确保所有接口错误返回符合 `04-api-spec.md`。
- **实施**:
  - 检查并修改手动构建的错误响应。
  - 格式统一为 `{ success: false, error: { code: 'ERR_CODE', message: '...' } }`。

## 阶段二：核心逻辑对齐需求 (Quality Fix)
### 1. 修复工时计算逻辑
- **目标**: 落实 `SW63` 及需求分析中关于“休息时间扣除”的要求。
- **文件**: `packages/server/src/modules/attendance/domain/attendance-calculator.ts`
- **实施**:
  - 在 `calculate` 方法中引入 `restStartTime` 和 `restEndTime` 处理。
  - 计算打卡区间与休息区间的交集时长，并在总工时中扣除。
  - 确保扣除后的工时非负。

## 阶段三：测试与验证 (Verification)
### 1. 单元测试补充
- **文件**: `packages/server/src/modules/attendance/domain/attendance-calculator.test.ts`
- **用例**:
  - `should deduct rest time when shift includes break`: 标准午休扣除测试。
  - `should not deduct rest time when work is outside break`: 加班/非午休时段测试。

### 2. 接口测试验证
- **验证**: 调用 `/recalculate` 接口，验证异常情况下的 JSON 格式及日志输出。
