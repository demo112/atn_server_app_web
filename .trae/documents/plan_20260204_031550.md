我已完成对报错日志的深入分析。问题根源在于 Server 端对输入参数的运行时校验不足（导致 500 错误）以及 Web 端 Schema 定义过于严格（导致 ZodError）。

以下是针对性的修复计划：

### 1. Server 端修复（解决 500 错误）
**目标**：防止因非法参数（如 `page=NaN`）导致 Prisma 崩溃。
- **Statistics 模块**：
  - 修改 `StatisticsController`，在调用 Service 前强制校验 `year`/`month` 参数，非数字直接返回 400。
- **Attendance 模块**：
  - 修改 `AttendanceClockService` 和 `AttendanceCorrectionController`，对 `page`、`pageSize`、`employeeId` 等参数进行安全的数字转换，处理 `NaN` 情况。
  - 修正 `AttendanceSettingsController`，增加对 `day_switch_time` 的格式校验。

### 2. Web 端修复（解决 ZodError）
**目标**：确保前端 Schema 能够正确解析 Server 返回的数据。
- **Clock Records**：
  - 检查并修正 `PaginatedClockRecordSchema`，确保分页字段结构与 Server 返回的扁平/嵌套结构一致。
  - 放宽 `ClockRecordSchema` 中可能为空字段的校验（如 `location`、`deviceInfo`）。
- **Leaves & Employees**：
  - 修正 `LeaveVoSchema`，确保 `status` 和 `type` 枚举值与后端完全匹配。
  - 确保所有 ID 字段兼容 `string` 类型（后端 BigInt 序列化）。

### 3. 补充测试（防止回归）
- **新增 Server 测试用例**：
  - 模拟非法参数（如字符串作为 ID、负数分页），验证 API 是否正确返回 400 而非 500。
  - 验证 `StatisticsService` 的缺失测试路径。

确认后，我将按照 **Server 修复 -> Web 修复 -> 测试验证** 的顺序执行。
