# 关于 5 条 500 错误日志的深度评估

您提到的“5 条日志”应指日志中出现的 5 次 **`AxiosError: Request failed with status code 500`**，它们分别出现在：

1. `CorrectionPage` (补卡)
2. `AttendanceSettingsPage` (设置)
3. `DailyRecords` (每日记录)
4. `SummaryPage` (统计汇总)
5. `ReportPage` (报表)

针对这类**后端服务崩溃**导致的错误，ENG01 体系的发现能力如下：

## 1. 发现能力评估

| 阶段                |     发现能力    | 详细分析                                                                                                                                                                |
| :---------------- | :---------: | :------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **开发与构建 (CI/CD)** | 🔴 **无法发现** | **原因：Mock 隔离**ENG01 的设计 (Story 2.2) 要求 Web 端测试使用 MSW 进行 Mock。这意味着测试运行时，前端**并不连接真实后端**，而是接收预设好的“成功数据”。**结果**：即使后端接口全部报 500，前端的单元/集成测试依然会**全部通过 (Pass)**，导致 Bug 逃逸上线。 |
| **运行时 (Runtime)** | 🟢 **可以发现** | **原因：日志治理**日志显示 `[error] [ERROR]` 及详细堆栈，证明 ENG01 的 **Logger 模块** 正在工作。它成功捕获了 Axios 的异常，没有让错误“吞没”在 Promise 中。**价值**：它提供了错误发生的**准确组件位置**和**文件行号**，这是 ENG01 治理带来的显著收益。 |

## 2. 结论

**ENG01 成功让这些 Bug “可见”（可排查），但未能“拦截”（可预防）。**

这暴露了 ENG01 当前阶段 (Web Coverage S1) 的一个主要盲区：**缺乏对真实后端服务的连通性验证**。

## 3. 补强建议

为了在 CI 或部署阶段拦截这 5 个 500 错误，建议在 ENG01 中补充以下环节：

1. **S2 阶段：后端契约测试** (验证接口定义一致性，减少 400)。
2. **S3 阶段：端到端 (E2E) 冒烟测试** (使用 Cypress/Playwright)。

   * 只需编写 1 个测试用例：`登录 -> 访问 Dashboard -> 加载统计数据`。

   * 配置其连接**真实测试环境后端**。

   * 一旦后端报 500，测试立即失败，阻止上线。
